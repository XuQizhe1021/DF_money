# 阶段4性能优化报告（M4）

## 1. 目标

在不做破坏性重构的前提下，优先完成可验证的小步优化，降低链路时延、请求放大和数据库压力。

## 2. 优化项与对比证据

### 2.1 看板涨跌榜请求聚合

- 优化前：前端为计算7日涨跌榜，最多并发请求 12 次 `/api/ammo/{id}/history`。
- 优化后：改为 1 次 `/api/ammo/change-ranking?days=7&limit=3` 聚合查询。
- 对比指标：接口请求数 `12 -> 1`，请求降幅约 `91.7%`。
- 影响：弱网环境下看板首屏稳定性与可用性明显提升。

### 2.2 持仓列表查询路径优化

- 优化前：持仓列表逐条调用最新价查询，存在N+1查询放大。
- 优化后：改为一次批量最新价映射，单次列表查询固定为“列表+价格映射”两段查询。
- 对比指标：数据库查询复杂度由 `1 + N` 收敛为 `2`（忽略分页参数校验）。
- 影响：持仓条目增长时，接口响应时间增长斜率显著下降。

### 2.3 热点接口短TTL缓存

- 优化前：`/api/ammo/latest` 高频轮询时每次都触发相同SQL。
- 优化后：后端增加5秒短TTL内存缓存并在写入时主动失效。
- 对比指标：同一窗口内重复查询命中缓存，数据库查询次数趋近 `K -> 1`（K为5秒内请求次数）。
- 影响：在前端频繁刷新与多页面切换场景下，后端更稳定。

### 2.4 图表渲染与交互优化

- 优化前：快速切换子弹/时间窗口会重复触发请求与图表重绘。
- 优化后：增加30秒历史数据缓存 + 180ms防抖；ECharts启用 `showSymbol=false`、`sampling=lttb`、`animation=false`、长序列 `dataZoom`。
- 对比指标：高频切换时重复请求次数显著下降，长序列渲染卡顿降低。
- 影响：图表交互连贯性提升，页面抖动减少。

## 3. 性能相关接口变更

- 新增：`GET /api/ammo/change-ranking`
- 扩展：`GET /api/holdings` 支持 `limit`、`offset`
- 扩展：`GET /api/alerts/events` 支持 `offset`

## 4. 结论

本阶段性能优化以“请求数下降、查询路径收敛、渲染负担降低”为主，均为可回滚的小步改动，且已通过全量自动化测试验证。
